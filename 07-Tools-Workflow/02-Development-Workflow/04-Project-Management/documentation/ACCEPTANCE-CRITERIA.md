# Acceptance Criteria

Acceptance criteria define when a user story is complete. This guide covers writing effective acceptance criteria and best practices.

## Acceptance Criteria Overview

Acceptance criteria:
- **Definition**: Define completion
- **Testable**: Testable conditions
- **Clear**: Clear and specific
- **Agreed Upon**: Team agreement

## Criteria Format

### Given-When-Then

```
Given [initial context]
When [action]
Then [expected outcome]
```

### Example

```
Given a user is logged in
When the user clicks "Change Password"
Then the password change form is displayed
```

## Criteria Types

### Functional Criteria

- **Functionality**: What should work
- **Behavior**: Expected behavior
- **Interactions**: User interactions
- **Outcomes**: Expected outcomes

### Non-Functional Criteria

- **Performance**: Performance requirements
- **Security**: Security requirements
- **Usability**: Usability requirements
- **Compatibility**: Compatibility requirements

## Writing Criteria

### Best Practices

1. **Specific**: Be specific
2. **Testable**: Must be testable
3. **Clear**: Clear and understandable
4. **Complete**: Complete coverage
5. **Agreed**: Team agreement

### Common Patterns

- **Given-When-Then**: BDD format
- **Checklist**: Simple checklist
- **Scenarios**: Multiple scenarios
- **Conditions**: Conditional criteria

## Best Practices

### Criteria Development

1. **Collaborative**: Write collaboratively
2. **Examples**: Include examples
3. **Edge Cases**: Consider edge cases
4. **Review**: Regular review
5. **Update**: Update as needed

### Criteria Usage

1. **Development Guide**: Guide development
2. **Testing**: Use for testing
3. **Review**: Review against criteria
4. **Acceptance**: Acceptance verification
5. **Documentation**: Documentation
