# Sprint Reviews

Sprint Review (Sprint Demo) demonstrates work completed during the sprint. This guide covers sprint review purpose, format, and best practices.

## Sprint Review Overview

Sprint Review purpose:
- **Demonstrate Work**: Show completed work
- **Gather Feedback**: Get stakeholder feedback
- **Update Backlog**: Update product backlog
- **Collaboration**: Collaborate with stakeholders
- **Transparency**: Show transparency

## Sprint Review Format

### Typical Agenda

1. **Sprint Summary**: Brief sprint summary
2. **Demo**: Demonstrate completed work
3. **Feedback**: Gather stakeholder feedback
4. **Backlog Update**: Update product backlog
5. **Next Steps**: Discuss next steps

### Duration

- **2-week Sprint**: 1-2 hours
- **1-week Sprint**: 30-60 minutes
- **Time-boxed**: Stick to time limit

## Participants

### Attendees

- **Development Team**: Presents work
- **Product Owner**: Facilitates
- **Stakeholders**: Provide feedback
- **Scrum Master**: Supports team

### Preparation

- **Demo Preparation**: Prepare demo
- **Environment**: Set up demo environment
- **Backlog Review**: Review backlog items
- **Invitations**: Invite stakeholders

## Demo Best Practices

### Effective Demos

- **Show Working Software**: Demonstrate real features
- **User Stories**: Show user stories completed
- **Sprint Goal**: Relate to sprint goal
- **Interactive**: Make it interactive

### Demo Tips

- **Practice**: Practice demo beforehand
- **Keep Simple**: Keep it simple
- **Focus Value**: Focus on value delivered
- **Handle Questions**: Be ready for questions

## Feedback Collection

### Gathering Feedback

- **Ask Questions**: Ask for feedback
- **Listen**: Listen actively
- **Document**: Document feedback
- **Prioritize**: Prioritize feedback

### Feedback Types

- **Feature Requests**: New feature ideas
- **Improvements**: Suggestions for improvement
- **Bugs**: Issues found
- **Questions**: Questions about features

## Backlog Updates

### Updating Backlog

- **New Items**: Add new items from feedback
- **Prioritize**: Re-prioritize items
- **Refine**: Refine items
- **Plan**: Plan for next sprint

## Common Issues

### Problems to Avoid

- **Status Report**: Don't make it a status report
- **Too Technical**: Avoid technical jargon
- **No Demo**: Always show working software
- **Defensive**: Don't be defensive about feedback

### Solutions

- **Focus Demo**: Focus on demonstration
- **User Language**: Use user language
- **Show Value**: Show value delivered
- **Accept Feedback**: Accept feedback graciously

## Remote Reviews

### Virtual Reviews

- **Screen Share**: Share screen for demo
- **Recording**: Record for absent stakeholders
- **Interactive**: Keep it interactive
- **Tools**: Use collaboration tools

## Benefits

### Team Benefits

- **Recognition**: Team gets recognition
- **Feedback**: Immediate feedback
- **Alignment**: Stakeholder alignment
- **Motivation**: Team motivation
