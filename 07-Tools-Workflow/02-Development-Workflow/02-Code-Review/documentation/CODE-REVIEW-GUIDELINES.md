# Code Review Guidelines

Code review guidelines help ensure consistent, effective reviews. This guide covers guidelines for reviewers, authors, and teams.

## Reviewer Guidelines

### Review Approach

- **Be Constructive**: Provide helpful feedback
- **Be Respectful**: Respectful communication
- **Explain Why**: Explain your reasoning
- **Suggest Alternatives**: Suggest improvements
- **Acknowledge Good Code**: Recognize good work

### Review Focus

- **Functionality**: Does it work correctly?
- **Code Quality**: Is code well-written?
- **Tests**: Are there adequate tests?
- **Documentation**: Is it documented?
- **Standards**: Follows coding standards?

### Communication

- **Clear Language**: Use clear language
- **Specific**: Be specific about issues
- **Actionable**: Provide actionable feedback
- **Positive Tone**: Maintain positive tone

## Author Guidelines

### PR Preparation

- **Small PRs**: Keep PRs small and focused
- **Clear Description**: Write clear PR description
- **Tests**: Include tests
- **Documentation**: Update documentation
- **Self-Review**: Review your own code first

### Responding to Feedback

- **Be Open**: Be open to feedback
- **Ask Questions**: Ask if unclear
- **Address Feedback**: Address all feedback
- **Thank Reviewers**: Thank reviewers
- **Learn**: Learn from feedback

## Team Guidelines

### Review Process

- **Review Time**: Set review time expectations
- **Reviewers**: Assign appropriate reviewers
- **Review Size**: Limit review size
- **Review Frequency**: Regular review schedule

### Standards

- **Coding Standards**: Define coding standards
- **Review Checklist**: Create review checklist
- **Documentation**: Document review process
- **Training**: Train team on review process

## Review Checklist

### Functionality

- [ ] Code works as intended
- [ ] Edge cases handled
- [ ] Error handling present
- [ ] Performance acceptable

### Code Quality

- [ ] Code is readable
- [ ] Code follows standards
- [ ] No code smells
- [ ] Proper abstractions

### Tests

- [ ] Unit tests included
- [ ] Integration tests if needed
- [ ] Test coverage adequate
- [ ] Tests are meaningful

### Documentation

- [ ] Code is documented
- [ ] Comments explain why
- [ ] README updated if needed
- [ ] API documentation updated

## Common Patterns

### Good Reviews

- **Specific Feedback**: Specific, actionable feedback
- **Explanations**: Explain reasoning
- [ ] **Suggestions**: Suggest improvements
- **Positive Reinforcement**: Acknowledge good work

### Bad Reviews

- **Vague Comments**: "This is bad"
- **Personal Attacks**: Attacking person
- **Nitpicking**: Minor style issues
- **Blocking for Minor Issues**: Blocking for trivial issues

## Best Practices

### Review Timing

- **Prompt Reviews**: Review within 24 hours
- **Regular Reviews**: Review regularly
- **Avoid Rush**: Don't rush reviews
- **Schedule Time**: Schedule review time

### Review Size

- **Small PRs**: Keep PRs under 400 lines
- **Focused Changes**: One concern per PR
- **Incremental**: Make incremental changes
- **Reviewable**: Make PRs reviewable
